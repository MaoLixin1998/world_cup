# 世界杯 RAG MVP 设计文档

## 状态

本设计于 2026-06-10 经确认后写入文档。

在你明确确认本设计文档之前，不编写任何实现代码。

## 产品目标

构建一个单机优先的世界杯 RAG 应用，面向普通球迷。MVP 要让球迷可以用中文提问世界杯相关问题，浏览核心资料，查看来源链接和资料更新时间，并使用一个轻量球迷身份，而不是完整登录系统。

## 目标用户

MVP 的核心用户是普通球迷。

体验优先级：

- 回答快、好懂。
- 来源链接清楚。
- 界面和回答中文优先。
- 能轻松浏览球队、比赛、球员/人物、场馆和世界杯历史资料。
- 对 2026 相关内容明确说明时效边界。

## MVP 内容范围

MVP 覆盖：

- 世界杯历史资料。
- 2026 世界杯基础资料。
- 已进行赛程记录。
- 稳定公开参考资料。
- 用户提供且自有或有权使用的资料。

MVP 不承诺：

- 实时比分。
- 突发新闻。
- 伤病动态。
- 临场阵容。
- 实时战术解说。

所有面向用户的数据页面都应展示相关资料的最后更新时间。

## 版权与来源策略

MVP 采用保守的“摘要优先”策略：

- 回答用中文总结来源材料。
- 回答包含来源标题和链接。
- 产品不展示大段来源原文。
- 当知识库缺少可靠材料时，回答应明确说明资料不足，而不是编造。
- 来源元数据尽量包含 URL、标题、来源类型、语言、抓取或导入时间、更新时间。

## 时效性策略

MVP 使用每日定时更新。

每日更新任务可以抓取稳定公开资料，也可以处理用户提供的资料。任务状态、最近成功更新时间和错误摘要需要在管理页可见。

架构应允许未来从每日更新升级到更高频更新，但 MVP 不包含实时基础设施。

## 仓库与部署策略

使用一个 monorepo：

```text
world_cup/
  apps/
    backend-java/
    rag-python/
    web-react/
    web-vue/
  packages/
    api-contracts/
  infra/
    docker-compose.yml
    postgres/
    nginx/
  docs/
    superpowers/
```

MVP 只实现 `web-react`。`web-vue` 预留给后续功能等价的 Vue 3 前端，并复用同一套 API 契约。

部署必须单机优先，使用 Docker Compose。MVP 避免 Kubernetes、Kafka、Milvus、Elasticsearch 和其他重型基础设施。

## 技术选择

- Java 主业务后端：Spring Boot 3 + Java 21。
- Python RAG、数据处理与评测服务：FastAPI + Python 3.12。
- 前端：React + TypeScript + Vite 先行。
- 未来第二前端：Vue 3 + TypeScript + Vite。
- 数据库与向量检索：PostgreSQL + pgvector。
- LLM 与 embedding：通过 provider 抽象调用外部 API。
- 未来可选服务：Go 网关或实时推送服务，仅在出现真实需求后加入。

## 中国大陆开发环境约束

开发者默认在中国大陆网络环境下工作。

任何软件安装、依赖安装或镜像拉取前，都要先检查国内镜像源、代理或加速配置是否需要启用。包括但不限于：

- Homebrew。
- npm。
- pip。
- Gradle/Maven。
- Docker 镜像。
- Python、Java、Node.js 相关工具链。

如果使用官方默认源，需要确认当前环境访问正常，或由用户明确要求使用官方源。

## 契约、注释与日志规范

OpenAPI 契约中的 `summary` 和 `description` 使用中文，方便中文 review 和自动生成中文接口文档。

实现代码需要有充足但不啰嗦的中文注释：

- 简单 getter、setter、普通结构声明不需要注释。
- 复杂业务规则、边界判断、RAG 流程、调度流程、部署配置需要中文解释性注释。
- 注释解释“为什么这样做”和“边界是什么”，避免重复代码字面含义。

关键节点需要日志：

- Java 后端记录匿名用户创建、聊天请求开始/结束、RAG 调用成功/失败、管理操作和异常。
- Python RAG 服务记录导入任务开始/结束、来源数量、chunk 数量、embedding 批次、检索命中、无可靠来源 fallback 和评测结果。
- 前端记录关键用户动作时应优先使用可观测埋点接口或调试日志，不在生产控制台输出敏感信息。

日志文案优先中文。日志不得记录密钥、token、个人敏感信息，也不得记录大段用户问题原文或来源原文。

## 运行时架构

```text
浏览器
  -> React 前端
  -> Spring Boot API
      -> PostgreSQL + pgvector
      -> FastAPI RAG 服务
            -> 外部 LLM API
            -> 外部 embedding API
            -> PostgreSQL + pgvector
```

前端默认只调用 Java API。Java 服务负责球迷身份、会话、聊天记录持久化、浏览 API 和管理 API。Python 服务负责导入、清洗、切块、embedding、检索、回答生成和 RAG 评测。

PostgreSQL + pgvector 是 MVP 唯一核心状态层。

## 核心数据模型

### 匿名球迷身份

`fan_user`

- `uid`
- `nickname`
- `avatar_player_id`
- `created_at`
- `last_seen_at`

应用没有密码登录。用户首次打开时创建轻量球迷身份，包括 uid、昵称和选择的明星球员头像。后端保存这条匿名用户记录。MVP 的找回能力仅限于用户自己记得 uid。

### 世界杯领域数据

`tournament`

- 年份
- 主办地
- 阶段元数据
- 简介

`team`

- 名称
- 国家或地区
- 别名
- 队徽元数据，如果可用
- 简介

`player`

- 姓名
- 所属国家队或相关球队
- 头像元数据
- 简介
- 是否是明星头像候选

`match`

- 赛事年份
- 阶段
- 主队或 A 队
- 客队或 B 队
- 计划时间
- 场馆
- 是否已进行
- 来源资料中存在时可保存比分字段，但 UI 不承诺实时比分

`venue`

- 名称
- 城市
- 容量，如果可用
- 简介

### RAG 数据

`source_document`

- 标题
- URL
- 来源类型
- 语言
- 许可或权利说明，如果已知
- 抓取或导入时间
- 更新时间

`document_chunk`

- 来源文档 id
- chunk 摘要或规范化文本
- 实体标签
- 时间范围标签
- 来源引用元数据

`document_embedding`

- chunk id
- 向量
- embedding 模型
- 创建时间

### 聊天与反馈数据

`chat_session`

- uid
- 标题
- 创建时间

`chat_message`

- 会话 id
- 角色
- 内容
- 引用来源
- 延迟毫秒数

`answer_feedback`

- 消息 id
- 是否有帮助
- 原因

### 导入与评测数据

`ingest_job`

- 状态
- 开始时间
- 结束时间
- 来源数量
- chunk 数量
- 错误摘要

`rag_eval_case`

- 问题
- 期望回答要点
- 期望引用提示
- 分类

`rag_eval_run`

- 检索分数
- 引用覆盖率
- 回答分数
- 延迟

RAG 评测在 MVP 中是工程测试或 CLI 能力，不做成用户页面。

## RAG 问答流程

```text
React 问答页
 -> Java /api/chat
 -> Java 保存用户消息
 -> Java 调用 FastAPI /rag/answer
 -> FastAPI 规范化中文问题
 -> FastAPI 做向量检索和元数据过滤
 -> FastAPI 可选执行轻量重排
 -> FastAPI 生成中文回答
 -> FastAPI 返回回答、引用、相关实体和更新时间
 -> Java 保存助手消息
 -> React 展示回答、来源链接、资料新鲜度和相关浏览入口
```

回答规则：

- 优先使用简洁、球迷友好的中文。
- 事实性回答需要包含来源链接。
- 展示资料更新时间。
- 避免无依据断言。
- 如果可靠上下文不足，要说明知识库没有足够可靠资料。

## 每日导入流程

```text
调度器
 -> FastAPI 导入流水线
 -> 抓取稳定公开来源并读取用户提供资料
 -> 清洗和规范化记录
 -> 抽取或关联实体
 -> 文档切块
 -> 创建 embedding
 -> upsert 到 PostgreSQL 和 pgvector
 -> 写入 ingest_job 状态
 -> Java 管理 API 读取状态并展示给前端
```

调度器可以内置在 Python 服务中，也可以是一个轻量 Compose 服务。它不应依赖 Airflow、Kafka、Kubernetes 或其他重组件。

## UI 页面

### 球迷首页

首次打开：

- uid 输入。
- 昵称输入。
- 明星球员头像选择。

再次访问：

- 欢迎语。
- 世界杯提问入口。
- 浏览入口。
- 热门问题提示。
- 最后更新时间。

审查标准：

- 所有输入都有可见 label。
- 头像触控目标至少 44 px。
- 移动端布局不拥挤。
- 更新时间可见，但不喧宾夺主。

### 问答页

问答页提供：

- 世界杯问题输入。
- 示例问题。
- 回答卡片。
- 来源链接。
- 资料更新时间。
- 相关球队、比赛、球员或场馆链接。

审查标准：

- 引用来源可点击。
- 无来源回答有明确标记。
- 加载和错误状态清楚。
- 长答案在移动端仍然可读。
- 输入区域不遮挡对话内容。

### 浏览页

浏览页提供以下分区：

- 球队。
- 比赛。
- 球员和人物。
- 场馆。

应支持搜索和简单筛选。详情页应展示核心事实、相关比赛、来源新鲜度，并提供继续追问入口。

审查标准：

- 列表易扫描。
- 比赛行能清楚比较阶段、球队、时间和状态。
- 状态不只靠颜色表达。
- 移动端和桌面端布局稳定，避免横向滚动。

### 我的记录

我的记录页提供：

- uid。
- 昵称。
- 头像。
- 提问历史。
- 有价值时展示回答反馈历史。
- 继续会话动作。

审查标准：

- 历史记录排序清楚。
- 空状态包含有用行动入口。
- 文案不误导用户以为这是安全账号系统。

### 管理页

管理页使用环境密钥或 Basic Auth 保护。管理页提供：

- 数据源状态。
- 最近成功更新时间。
- 最新导入任务。
- 触发重建索引动作。
- 错误摘要。

审查标准：

- 重建类操作需要确认。
- 运行中的任务显示进度或清晰状态。
- 错误信息可读。
- 不暴露密钥。

## 全局 UI/UX 标准

每个实现完成的页面都必须经过 ui-ux-pro-max 审查。

全局标准：

- 中文优先。
- 像体育内容产品，而不是营销落地页。
- 首屏直接可用。
- 普通文本满足 WCAG AA 对比度。
- 键盘焦点状态可见。
- 重要控件触控目标至少 44 px。
- 图标使用一致的 SVG 图标库，例如 Lucide。
- 不用 emoji 作为功能图标。
- 响应式布局稳定，避免意外横向滚动。
- 加载状态预留空间，减少布局跳动。
- 内容页清楚展示资料更新时间和来源。
- 配色克制，可使用球场绿、深墨色、白色和少量金色强调，但避免整站变成单一绿色主题。

## 工程流程

实现阶段遵循 TDD：

- 先写失败测试。
- 运行测试，确认它因为预期原因失败。
- 实现最小可用行为。
- 运行测试和验证。
- 做小而聚焦的提交。

每个 UI 页面完成前，都必须通过 ui-ux-pro-max 的体育内容产品体验审查。

React 先实现。Vue 在 React MVP 稳定后，基于同一 API 契约再实现。

## 已确认边界

- MVP 使用一个 GitHub 仓库。
- MVP 部署在一台小服务器上。
- MVP 使用 Docker Compose。
- MVP 不使用 Kubernetes、Kafka、Milvus 或 Elasticsearch。
- MVP 使用外部 LLM 和 embedding API。
- MVP 把匿名球迷记录保存在后端数据库中。
- MVP 不包含完整登录、密码账号、邮箱、手机号验证或社交登录。
